## 三、对照原始本源目标 · 逐条辨析

### 目标①: 「彻底脱离对Agent的依赖」

| 操作 | 是否需要Agent | ACU成本 | 判定 |
|---|---|---|---|
| 创建VM | **是** (session/prompt) | ~$0.50 | **不符** |
| 隧道续期 | **是** (session/prompt) | ~$0.10-0.50 | **不符** |
| 直接HTTP操作 | **否** (/_/run等) | 0 | **符合** |
| 读文件 | **否** (/_/file) | 0 | **符合** |
| 写文件 | **否** (PUT /_/file) | 0 | **符合** |
| 启后台进程 | **否** (/_/spawn) | 0 | **符合** |
| 发送Agent消息 | 404/405 | N/A | **不可用** |
| 读Agent输出 | 404 | N/A | **不可用** |

**结论**: 创建和续期必须经Agent · 仅运行态真正脱离 · **部分符合**

### 目标②: 「不消耗金额」

```
初次创建: 208 VMs × ~$0.50 = ~$104
隧道续期: 每次 ~$0.10-0.50/VM
保活成本: WSS ping = 0 ACU · HTTP ping = 0 ACU
直接操作: 0 ACU ✓

实际已消耗: deep_audit发现32账号out_of_quota
余额范围: $-0.07 ~ $0.00 (已耗尽)
VM活跃余额: $-0.00 ~ $96.50

结论: 初次创建+续期有成本 · 运行态0成本 · **不完全符合**
```

### 目标③: 「无限制并发」

```
硬性限制:
  - 每账号 ≥1 session (实证: 3session/账号可行)
  - session/new rate limit存在
  - out_of_quota阻断新建
  - 当前: 208 VM × 0 alive = 0 可用并发

理论上限:
  - 200账号 × 3 session = 600 VM
  - 但实际受tunnel存活影响

结论: 理论可扩展 · 实际受多重限制 · **不完全符合**
```

### 目标④: 「VM自治 (无人干预永续)」

```
当前现实:
  - VM创建后 → 依赖本地vm_omni.js进程保活
  - 本地进程退出 → WSS断 → tunnel ~30min死亡
  - tunnel死 → VM不可达 → 无法自愈
  - session过期 → 无法重连 → 彻底死亡

自治缺失环节:
  ✗ VM内无自守护cloudflared (vm_daemon仅管dao_proxy)
  ✗ 无VM→外部的反向心跳机制
  ✗ 无token自刷新
  ✗ 无tunnel自恢复
  ✗ 208个VM全灭 = 设计未实现自治

结论: **严重不符**
```

## 四、根本矛盾分析

### 矛盾一: 短命隧道 vs 永续目标

```
trycloudflare.com: 临时快穿 · 无认证 · DNS ~30min-6hr自动注销
devinapps.com: Devin官方expose · 需Agent调deploy工具创建
两者共同点: 都是一次性的 · 不可自行续命

vm_omni.js虽有HTTP keepalive (2min/次)
  → 但仅在本地进程存活时生效
  → 本地进程退出 = keepalive停 = tunnel必死

根本问题: 没有VM侧的自守护tunnel机制
```

### 矛盾二: Agent是创造者也是瓶颈

```
创建VM → 必须Agent执行bash → 消耗ACU → 受quota限制
续期tunnel → 必须Agent执行RENEWAL_BASH → 消耗ACU
Agent通信 → 仅ACP WSS可用 · REST API全404

paradox:
  要「脱离Agent」→ 但Agent是唯一能在VM上执行命令的途径 (在tunnel死后)
  tunnel活着 → 可经HTTP直操作 · 不需Agent (0 ACU)
  tunnel死了 → 只能经Agent重建 · 必需Agent (消耗ACU)
  → 自治 = tunnel永不死 · 但tunnel物理上必死
```

### 矛盾三: 两套隧道系统混战

```
vm_omni.js创建:
  → Agent调deploy → devinapps.com URL (2个)
  → 官方管理 · 有认证 (user:password@host)
  → 但不可外部续期

RENEWAL_BASH创建:
  → 自装cloudflared → trycloudflare.com URL (34个)
  → 无认证 · 无身份 · 纯临时
  → 依赖cloudflared进程存活

两套系统不兼容:
  - devinapps有密码 → RENEWAL_BASH不产生密码
  - trycloudflare无密码 → vm_omni原始记录有密码
  - 续期后URL类型变了 → 认证方式不一致
```

### 矛盾四: 凭证断裂

```
dao_benyuan_state: auth1_xxx (195个 · 新鲜)
vm_eternal_pool: devin-session-token$eyJ... (208个 · 全死)

断裂点:
  - nucleus.js loadAccounts() 从 accounts.json 读 apiKey
  - 但 apiKey 格式是 "devin-session-token$xxx" (已过期的session token)
  - 而非 "auth1_xxx" (可用于创建新session的认证token)
  - benyuan_state的auth1_xxx 从未被nucleus.js使用

修复路径:
  1. 用auth1_xxx重新PostAuth → 获得新session_token
  2. 用新session_token连接ACP → session/new → 新VM
  但这等于重新创建全部208个VM · 消耗 ~$104
```

## 五、对照《阴符经》· 道之审视

### 「火生於木，禍發必尅」

当前系统的"木"(根基)是:
- 短命的trycloudflare隧道
- 必须Agent参与的创建/续期
- 一次性的session token

"火"(灾难)必然发生:
- 隧道死亡 → 全灭
- Token过期 → 不可恢复
- Agent quota耗尽 → 无法续命

**评**: 架构根基(木)本身含"火"种 · 灾必发 · 今已应验

### 「三盜既宜，三才既安」

三盗 = 天(Devin平台)盗人(使用者)· 人盗万物(VM资源)· 万物盗天(计算回馈)

当前失衡:
- 天(Devin)不断回收: session过期 · tunnel销毁 · quota消耗
- 人(使用者)不断索取: 208 VM · 0成本期望 · 永续期望
- 万物(VM)无法自立: 不能自守护 · 不能自续命

**评**: 三盗不宜 · 故三才不安 · 系统必溃

### 「至靜之道，律曆所不能契」

真正的"至静"应是:
- VM自身完备 · 不依赖外部保活
- 隧道自愈 · 不需要Agent重建
- Token自刷 · 不因过期而死

当前"动"太多:
- 每2min HTTP ping · 每25s WSS ping · 每10min守护循环 · 每15min续期扫描
- 所有"动"都在本地 · VM内无"静"

**评**: 以动求静 · 本末倒置

**「反者，道之動也。」**

当前的"正向"思路是: 本地管控 → Agent创建VM → 隧道暴露 → HTTP直操作。
但"反者"揭示: 系统的死亡正来自其创造方式本身 — 用短命的手段追求永续 · 用依赖Agent的方式追求脱离Agent。

**「生者死之根，死者生之根。」**

208个VM的"生"(创建) · 其中已含"死"(ephemeral tunnel + session expiry)。
唯有直面此根 — VM内部自完备 · tunnel自愈 · 凭证自刷 — 方可"死者生之根" · 从当前全灭中重生。
