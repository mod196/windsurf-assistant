## 二、五层解构本源 (反者道之动 · 不被表象束缚)

### 第一层 · 表象需求

"切号优先切到有额度的账号"

### 第二层 · 直接目标

"不让额度账号被冷落"

### 第三层 · 真正诉求

**「不浪费已付费/已分配的真金白银」**  
overage = $200 Extra Usage · 一次性资金 · 用完归零  
百分比 = 周期配额 · 每周/每日自动重置 · 等待即回潮

### 第四层 · 资源本质 (核心顿悟)

| 维度 | overage 美元 | 百分比配额 |
|---|---|---|
| 经济学性质 | **存量 (Stock)** | **流量 (Flow)** |
| 再生性 | 不可再生 · 用一分少一分 | 可循环 · 周期重置 |
| 不用的代价 | **沉没浪费** (废账户即损失) | 等待即回来 (无损失) |
| 道家映射 | "天下之物生于有" · 已生之物即损 | "有生于无" · 无穷归来 |

### 第五层 · 病灶的真正本源 (错误抽象)

**当前 `_scoreOf` 把两种本质不同的资源放在同一个连续分数坐标系里比大小** — 这是**错误的抽象**！  

道家曰「**天之至私，用之至公**」 — 不该混在一起算的东西，要分开各得其所。  
即「**九竅之邪，在乎三要，可以動靜**」 — 决策应分层：先选层 → 再选号。

## 三、病灶实证 (v3.2.1 之前)

### 3.1 评分公式

```javascript
// v3.2.1 之前的 _scoreOf
if (h.overageActive) {
  let s = 300 + Math.min(100, h.overageDollars || 0);  // ← 病灶1: min(100,$) 封顶
  if (h.staleMin < 15) s += 60;                          // ← 病灶2: 时效加成
  ...
  return s; // 区间 [150, 460]  ← 病灶3: 与百分比同坐标系
}
// 百分比模式
let s = Math.max(h.weekly, 0) * 8 + Math.max(h.daily, 0) * 3;
// 区间 [0, 1830]  ← 百分比上限远高于 overage 上限
```

### 3.2 分数对比表

| 账号类型 | 实际分数范围 | 主公图1 案例 |
|---|---|---|
| 💎 overage 账号 | 150 ~ 460 | $195/$189/$208/$193/$185 全部=**400 分** (被 min(100,$) 封顶) |
| 📊 百分比 W50/D50 | ~480 ~ 750 | 50×8 + 50×3 + 80 = **550~750 分** |
| 📊 百分比 W100/D100 | 最高 **1830 分** | — |

### 3.3 结论

> **任何 W50 百分比账号 ≥ 480 分 > 任何 overage 账号 ≤ 460 分**  
> 自动切号永远优先选百分比 → 额度账号被永久冷落 → 真金白银沉没浪费

═══════════════════════════════════════════════
│

└─ -∞   永禁 (无密码 / 用户主动锁 skipAutoSwitch)

### 4.2 自然顺应 (无为而无不为)

| 场景 | v3.3.0 行为 (自然产生 · 无需额外代码) |
|---|---|
| 全池有 overage 与百分比 | 优先选 overage 中 $ 最多者 |
| 当前 active = overage 切号 | 自然选下一 overage (excludeIdx 排自己) |
| 全部 overage 已用 (overageActive=false) | 自然下沉百分比层 |
| 重置时刻 overage 复活 | `_scheduleResetRefresh` 触发 verify → 自然上跃 |
| 用户锁了 overage 号 (skipAutoSwitch) | 跳过该号 · 下沉下一候选 |

### 4.3 软门控

`wam.preferOverageFirst` (默认 `true` · 道法自然 · 推荐)

| 值 | 行为 |
|---|---|
| `true` | 严格分层 · overage 绝对优先于百分比 (v3.3.0 默认 · 实现主公诉求) |
| `false` | 回退 v3.2.1 统一坐标系 · 兼容旧行为 |
