## 四 · 五辩五立 (辨证)

### 辩 ① · 何不是 LS 进程 crash？

LS 是独立子进程（spawn 自 Windsurf 主进程）。LS crash 时:
- ext-host 之 dao-proxy-min 仍活，可继续响应 webview
- 但主公诉「**内部所有模块基本重启**」——LS 死不致 dao-security / rt flow 重启
- **不契 ②**

### 辩 ② · 何不是 Windsurf 主进程 crash？

主进程死之症:
- 整个 Windsurf 窗口消失，需重启动 Windsurf
- 但主公诉「**不导致 windsurf 本体重启**」
- **不契 ①**

### 辩 ③ · 何不是 v9.9.27 onDidChange watchdog reloadWindow？

reloadWindow 之症:
- ext-host + renderer 全重 init
- 所有扩展 deactivate hook **正常触发**（normal exit）
- rt flow / dao-security 之 webview 重新 init，message bridge 重建
- **主公无需重 Windsurf 即可恢 rt flow**
- 但主公诉「**rt flow 前端无法使用 须重启 windsurf 才能恢复**」
- **不契 ⑤**

### 辩 ④ · 何不是 source.js 之 server.on("error") 之 process.exit(1)？

```js
// _runCli 内 L2923-2925
server.on("error", () => { process.exit(1); });
```

- 此钩仅装于 _runCli → ext-host require 路径下**未装**
- 模块顶层 server 之 error event 仅有 L2821 之 `server.on("error", (e) => log...)` 仅 log 不 exit
- **此路径在 ext-host 内安全**
- **不构成触发器**

### 辩 ⑤ · 何不是 dao-byok / dao-wanfa-guizong 之 NodeService crash？

- zhou 现态: 装 dao-proxy-min + dao-security · **无 dao-byok · 无 dao-wanfa-guizong** (印 158 §4 实测铁定)
- 而主公诉之症在 zhou 仍反复出现
- 故真根非 dao-byok / dao-wanfa-guizong（虽 INDEX VI 早期归因如此 · **误诊**）
- **真根唯在 dao-proxy-min · source.js · process.on 漏**
